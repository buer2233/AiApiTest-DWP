# project-info/demand/AGENTS.md

本目录是固定 loop 第 1 阶段：需求分析阶段。进入本目录工作前，必须先遵守根目录 `AGENTS.md` 和 `project-info/AGENTS.md`，再遵守本文件。

## 阶段目标

- 输出详细需求说明书，也就是概要设计。
- 明确需求背景、目标、范围、不做事项、用户角色、业务流程、功能清单、非功能要求、验收标准和变更记录。
- 同步完成功能对应的数据表设计，包括字段、关联关系、更新规则、状态流转和关键约束。
- 完成架构影响评估、API 契约草案和容器化兼容检查，为后续测试用例、UI 原型、后端和前端开发提供可并行的稳定输入。

## 需求深挖与可执行模板

- 必须使用 `_需求说明书模板.md`（可执行规格模板）作为需求说明书标准结构，复制为 `需求名-需求说明.md` 后按需求分级填充。
- 需求阶段采用「主导式深挖」：由 agent 主导逐项提问，把模板每个必填项问到无歧义，引导主人把需求挖透，而不是等主人一次写完。
- 系统性消灭下游脑补点：凡欠明确、有歧义或多解读的点，一律写入模板 §0 待澄清清单，逐条与主人确认，禁止在正文脑补关键决策。
- 冻结门禁：§0 全部闭环且架构影响评估、API 契约、容器化检查通过后，由主人在模板 §14 签字冻结，下游方可自动启动。
- 本阶段是自主流水线的杠杆支点——需求挖得越透，下游越能无人值守。整体执行方式见 `docs/自主开发流水线.md`。

## 技能要求

- 编写或修改需求分析资料时，优先使用 `product-requirements`。
- 新需求、新功能、架构或行为变更前，按根规则使用 `brainstorming` 做方案确认。
- 如果技能不可用，应记录原因，并在不跳过需求分析质量要求的前提下继续补齐文档。

## 文件命名

- 同一需求后续阶段必须沿用同一需求名，仅后缀不同。
- 推荐格式：`需求名-需求说明.md`。
- 草稿可使用 `需求草稿-主题.md`，但进入正式开发前应整理成正式需求说明书。

## 内容要求

- 每个功能必须说明能做什么、做到什么程度、满足什么要求。
- 每个功能都要关联对应的数据表或明确不涉及数据持久化。
- 表设计应至少包含表名、用途、字段、字段类型建议、必填性、默认值、索引建议、关联关系和状态流转。
- 明确与 `api-test/`、`jenkins/`、`back-end/`、`front-end/` 的职责边界。
- 验收口径必须能被 `project-info/test_case/` 转换为测试用例。
- 架构影响评估必须说明是否影响模块边界、数据模型、权限、Jenkins Job、`api-test` 执行协议、报告产物、Docker Compose 部署和安全策略。
- 涉及 DRF API 时必须给出 API 契约草案，包括路径、方法、请求参数、响应字段、错误码、分页筛选、权限要求和关键状态流转。
- 容器化兼容检查必须明确是否存在本机绝对路径、宿主机固定端口、真实凭据、不可迁移业务常量或手工 Jenkins 配置依赖。
- 需求说明书完成后，如果角色、页面目标、核心字段、状态流转、权限边界和验收口径已经明确，可以作为测试用例设计和 UI 原型设计并行启动的输入。

## 安全要求

- 需求应保持通用测试平台定位，不写入真实账号、密码、token、cookie、租户密钥、生产 URL 或敏感地址。
- 示例数据使用占位符或脱敏数据。
