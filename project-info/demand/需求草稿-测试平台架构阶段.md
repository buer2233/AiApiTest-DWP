# AI 自动化测试平台需求草稿（架构阶段）

## 文档定位

本文档用于临时沉淀架构讨论过程中已经明确的需求要点，避免在正式需求分析前丢失上下文。本文不是最终 PRD，不展开完整功能验收细节。

## 已确认架构方向

- 平台执行主干采用严格 Jenkins 模式：平台侧所有测试执行、模块重试、失败重试和报告生成都必须通过 Jenkins。
- DRF 后端不直接执行 pytest，不拼接 pytest 命令，不重复实现失败重试算法。
- `api-test` 是测试执行框架唯一实现位置；涉及 pytest 参数、失败 node id、Allure 产物、summary 和报告解析协议的变更，应优先在 `api-test` 内完成。
- Jenkins 负责 Pipeline、节点、环境、凭据、定时任务、队列、构建归档和 Allure 发布。
- DRF 后端负责平台数据建模、权限、审计、Jenkins 触发、Jenkins 状态同步、模块快照、失败用例记录和报告入口。

## 已确认功能草稿

### 每日全量执行

- 平台需要每天凌晨 1 点自动执行现有全部测试用例。
- 每日全量执行结果作为当天最新测试数据展示在平台首页或模块通过率页面。
- `api-test/test_case/` 下每个一级子目录视为一个测试模块。
- 每个模块需要展示模块名、用例包名、负责人、自动化负责人、通过率、运行日期、运行时间、成功数、失败数和错误数等信息。

### 模块重试

- 用户可以在平台上对单个模块发起模块重试。
- 模块重试需要通过独立 Jenkins Job 执行，重跑该模块目录下全部用例。
- 模块重试完成后更新该模块主表快照中的统计数据、运行日期和运行时间。
- 模块重试完成后，当前模块原有失败用例记录先标记为“不展示”，再写入本次执行产生的新失败用例。

### 失败重试

- 用户可以在平台上对单个模块发起失败重试。
- 失败重试需要通过独立 Jenkins Job 执行，只执行当前模块状态为“失败”的失败用例。
- 当模块通过率为 100% 时，平台不触发 Jenkins，并提示“通过率100%无需失败重试”。
- 失败重试完成后需要更新主表快照中的通过率、成功数、失败数和错误数。
- 失败重试不更新主表快照中的运行日期和运行时间。
- 失败重试执行通过的失败用例记录标记为“通过”。

### 模块互斥与并发

- 同一环境、同一模块在任意时刻只能有一个模块重试或失败重试任务处于排队、执行或取消中。
- 模块重试和失败重试互斥：同一模块执行其中一种重试时，不允许触发另一种重试。
- 模块重试 Job 和失败重试 Job 需要相互独立。
- 每类 Job 同时最多允许 10 个任务执行。

## 数据草稿

### 模块展示快照

- 表用途：固定保存所有模块的当前展示数据，每个环境下每个模块一条当前快照。
- 数据策略：不按每次执行新增主表行，而是按模块更新当前快照。
- 建议唯一约束：环境 + 模块标识。
- 典型字段：环境、模块标识、用例包名、模块名、负责人、自动化负责人、成功数、失败数、错误数、通过率、运行日期、运行时间、运行标记。

### 失败用例记录

- 表用途：保存失败用例记录，数据只新增或更新状态，不删除物理数据。
- 状态枚举：
  - `失败`：当前有效失败，会降低通过率。
  - `跳过`：用户手动跳过，不降低通过率。
  - `通过`：失败重试后已通过，不在当前失败列表中展示。
  - `不展示`：模块重试前历史失败记录归档，不在当前失败列表中展示。
- 失败用例页面只展示状态为“失败”和“跳过”的记录。
- 用户只能在“失败”和“跳过”之间手动切换状态。
- 通过率计算只扣减状态为“失败”的用例。
- 典型字段：模块标识、用例唯一标识、pytest node id、用例描述、错误类型、错误信息、执行结果、确认结果、关联缺陷、来源任务、最近重试任务、状态。

### Jenkins 执行记录

- 表用途：保存所有 Jenkins 执行记录，数据一直新增，不删除物理数据。
- 记录范围：每日全量、模块重试、失败重试、取消、标记失败等平台相关任务。
- 典型字段：任务名称、任务类型、环境、环境地址、模块标识、Jenkins Job 名称、队列 ID、构建号、构建 URL、报告 URL、执行状态、触发人、开始时间、结束时间。

## 建议表设计草案

> 本节是架构阶段的数据模型草案，用于约束后续 DRF 建模方向；正式字段类型、索引和迁移文件在后端设计阶段再细化。

### `test_environment`：测试环境表

| 字段 | 说明 |
| --- | --- |
| `id` | 平台内部环境 ID |
| `env_key` | 环境唯一标识，例如 `beta`、`release`、`staging` |
| `env_name` | 环境展示名称 |
| `env_type` | 环境类型，用于页面筛选和 Jenkins 参数 |
| `base_url` | 被测环境基础地址，正式配置应从环境变量或本地私有配置读取 |
| `is_active` | 是否启用 |
| `created_at` / `updated_at` | 创建和更新时间 |

设计说明：

- 用于解耦前端环境筛选、Jenkins 参数和模块快照环境归属。
- 不在仓库中固化真实生产地址或敏感地址。

### `test_module`：测试模块配置表

| 字段 | 说明 |
| --- | --- |
| `id` | 平台内部模块 ID |
| `module_key` | 模块唯一标识，建议与用例目录保持稳定映射 |
| `case_path` | pytest 模块路径，例如 `test_case/test_xxx_case` |
| `case_name` | 页面展示的用例包名 |
| `module_name` | 页面展示的模块名 |
| `dev_master` | 研发负责人 |
| `auto_master` | 自动化负责人 |
| `sort_order` | 页面排序 |
| `is_active` | 是否启用 |
| `created_at` / `updated_at` | 创建和更新时间 |

设计说明：

- `api-test/test_case/` 下每个一级子目录对应一个模块。
- `module_key` 是平台稳定标识，不建议直接依赖自增 ID 作为跨系统参数。
- 后续可以增加模块分组、标签、所属产品线等字段。

### `module_snapshot`：模块展示快照表

| 字段 | 说明 |
| --- | --- |
| `id` | 快照行 ID |
| `environment_id` | 关联测试环境 |
| `module_id` | 关联测试模块 |
| `case_external_id` | 兼容外部或历史接口的模块用例 ID |
| `pass_count` | 当前成功用例数 |
| `fail_count` | 当前有效失败用例数，只统计状态为“失败”的失败记录 |
| `error_count` | 当前错误用例数 |
| `total_count` | 当前模块总用例数 |
| `passed_percent` | 当前展示通过率 |
| `passrun_percent` | 当前执行通过率，后续正式定义 |
| `run_datetime` | 最近完整模块执行完成时间 |
| `run_time` | 最近完整模块执行耗时 |
| `run_mark` | 运行标记或备注 |
| `last_full_task_id` | 最近每日全量任务 |
| `last_module_rerun_task_id` | 最近模块重试任务 |
| `last_failed_rerun_task_id` | 最近失败重试任务 |
| `created_at` / `updated_at` | 创建和更新时间 |

建议约束：

- `environment_id + module_id` 唯一。

更新策略：

- 每日全量：更新全部模块快照，更新统计、`run_datetime` 和 `run_time`。
- 模块重试：更新当前模块快照，更新统计、`run_datetime` 和 `run_time`。
- 失败重试：更新当前模块快照的统计和通过率，不更新 `run_datetime` 和 `run_time`。
- 主表固定展示模块行，不按每次执行新增快照行。

### `failed_case`：失败用例记录表

| 字段 | 说明 |
| --- | --- |
| `id` | 失败记录 ID |
| `environment_id` | 关联测试环境 |
| `module_id` | 关联测试模块 |
| `case_external_id` | 兼容外部或历史接口的用例 ID |
| `node_id` | pytest node id |
| `case_info` | 用例说明或业务描述 |
| `case_desc` | 用例确认结果或补充描述 |
| `case_location` | pytest 用例定位信息 |
| `error_type` | 错误类型，例如断言失败、接口异常、环境异常 |
| `error_msg` | 错误详情，允许保存较长文本 |
| `result` | 原始执行结果 |
| `display_status` | 平台展示状态：失败、跳过、通过、不展示 |
| `bug_id` | 关联缺陷 ID |
| `source_task_id` | 产生该失败记录的 Jenkins 任务 |
| `last_retry_task_id` | 最近一次失败重试任务 |
| `occurred_at` | 失败发生时间 |
| `created_at` / `updated_at` | 创建和更新时间 |

状态规则：

- `失败`：默认状态，展示在失败用例列表，并扣减通过率。
- `跳过`：用户手动从失败切换，不扣减通过率，仍展示在失败用例列表。
- `通过`：失败重试后通过，不展示在当前失败用例列表。
- `不展示`：模块重试前将旧失败记录归档，不展示在当前失败用例列表。

操作规则：

- 失败记录不物理删除。
- 用户只能在“失败”和“跳过”之间手动切换。
- 模块重试写入新失败记录前，先将该模块当前“失败”和“跳过”的旧记录标记为“不展示”。
- 失败重试通过后，将对应失败记录标记为“通过”。

### `jenkins_task`：Jenkins 执行记录表

| 字段 | 说明 |
| --- | --- |
| `id` | 平台任务 ID |
| `task_name` | 任务展示名称 |
| `task_type` | 任务类型：每日全量、模块重试、失败重试 |
| `environment_id` | 关联测试环境 |
| `module_id` | 可选，模块级任务关联模块 |
| `case_path` | 本次执行的 pytest 路径 |
| `node_ids` | 本次失败重试执行的 node id 列表 |
| `job_name` | Jenkins Job 名称 |
| `queue_id` | Jenkins 队列 ID |
| `build_number` | Jenkins 构建号 |
| `job_url` | Jenkins 构建 URL |
| `report_url` | Allure 或归档报告入口 |
| `artifact_url` | 运行产物入口 |
| `status` | 平台任务状态 |
| `trigger_user_id` | 触发人 |
| `started_at` / `finished_at` | 开始和结束时间 |
| `created_at` / `updated_at` | 创建和更新时间 |

设计说明：

- Jenkins 任务记录只新增，不物理删除。
- 任务状态需要能覆盖排队、运行中、运行成功、运行失败、取消中、已取消、标记失败等状态。
- 前端的取消任务、查看报告、查看 Jenkins 任务、标记失败、查看进度等操作都基于该表和 Jenkins API。

### `module_execution_lock`：模块执行锁表

| 字段 | 说明 |
| --- | --- |
| `id` | 锁记录 ID |
| `environment_id` | 关联测试环境 |
| `module_id` | 关联测试模块 |
| `lock_type` | 锁类型，例如模块重试、失败重试 |
| `task_id` | 当前持锁任务 |
| `status` | 锁状态：持有、释放、过期 |
| `locked_at` / `released_at` | 加锁和释放时间 |

建议约束：

- 同一环境、同一模块同一时间只能存在一个有效锁。

设计说明：

- DRF 触发 Jenkins 前先尝试获取模块锁。
- Jenkins 任务结束、取消或标记失败后释放锁。
- 需要设计超时释放机制，避免 Jenkins 异常时模块永久锁死。

## 待后续正式需求分析确认

- 每日全量执行是否按模块拆分为多个 Jenkins 子任务，还是单个 Pipeline 内统一调度。
- 每日全量执行期间是否禁止人工重试。
- 通过率公式的最终字段定义需要统一。目前草稿按“只有状态为失败的用例扣减通过率”记录，后续需要用实际总用例数、成功数、失败数和错误数统一校验。
- Jenkins Job 命名规则、环境命名规则和模块标识规则。
- 失败用例唯一标识规则：使用 pytest node id、业务 case id，还是两者组合。
- Allure 报告与失败用例明细的解析协议。
