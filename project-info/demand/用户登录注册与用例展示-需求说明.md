## §1 需求背景与目标 [必填]

- **背景**：平台架构设计阶段（monorepo、严格 Jenkins 主干、DRF 后端、Vue 3 前端、Docker 化方案）已完成，但 `back-end/` 与 `front-end/` 目前为空（仅有规范文件），尚无任何可运行工程。平台缺少进入入口（认证）与最基本的"家底可见"（测试用例清单），执行主干 Jenkins 脚本也只有一个混合参数化 pipeline，与"模块重试 Job 与失败重试 Job 相互独立"的目标不符。
- **目标**：交付平台第一块地基，一次到位三件事——
  1. **人能进来**：用户可注册（待审批）→ 管理员审批 → 登录（DRF Token）→ 查看本人信息 / 登出。
  2. **看得见家底**：登录用户可浏览 `api-test/test_case/` 下的 pytest 用例清单（分页 + 筛选），数据由后端静态解析入库。
  3. **执行主干就位**：把 Jenkins 单一参数化 pipeline 重新设计为三个相互独立、可静态测试的 pipeline（每日全量 / 模块重试 / 失败重试），各自驱动 `api-test/tools/ci_runner.py`。
- **成功指标 / 价值**：
  - 端到端可演示：新用户注册 → admin 在 Django Admin 审批激活 → 登录拿到 token → 用例清单分页可见。
  - 后端 pytest 全绿且留存覆盖率证据；前端 Playwright 三页用例通过并留存截图；`jenkins/tests` 三个 pipeline 静态测试通过。
  - 为后续"模块通过率快照、失败用例、Jenkins 执行链路"需求铺好工程与执行主干地基。

## §2 范围 [必填]

**做（in scope）**

- 后端：Django + DRF + MySQL 工程脚手架；配置全部走环境变量。
- 后端 `accounts` app：自定义 User（含 `status`/`role`）、注册、登录（返回 Token）、登出、当前用户信息、Django Admin 审批入口。
- 后端 `testcases` app：用例同步接口（`ast` 静态解析 `api-test/test_case` → upsert 入快照表 → 消失用例软删）、用例列表接口（分页 + 筛选）。
- 前端：Vue 3 + Vite + TS + Element Plus + Vue Router + Pinia + Axios + TanStack Query 工程脚手架。
- 前端三页 + 基础布局：登录页、注册页、用例展示页；Axios 拦截器携带 DRF Token；路由守卫保护登录态；顶部显示当前用户 + 登出。
- Jenkins：把 `api-test-pipeline.groovy` 拆为三个独立 pipeline（`daily-full` / `module-rerun` / `failed-rerun`），各自调用 `ci_runner.py` 对应模式；补充/更新 `jenkins/tests` 静态测试。
- Docker：在 compose 中补 `backend` 服务雏形（连 `mysql` 服务、`API_TEST_ROOT` 卷挂载），保证容器化可迁移。

**不做（out of scope）**

- 模块通过率快照表、失败用例表、Jenkins 执行记录表、模块执行锁表（后续需求）。
- 前端触发执行/重跑、后端调用 Jenkins REST API、任务状态轮询与入库、报告解析入库（后续"执行链路"需求）。
- 前端用户管理 / 审批列表页（第一版审批走 Django Admin）。
- 密码找回、邮件通知、第三方登录、验证码、JWT / refresh token。
- 用例执行、用例编辑、用例与缺陷关联。
- 取消任务 job、标记失败 job 及其他 Jenkins job（后续需求）。
- 测试环境表与多环境切换（Jenkins 环境参数先以字符串参数占位）。

## §3 用户角色与权限矩阵 [必填]

> 第一版 `admin` 与 `member` 对**业务 API** 权限基本一致，但保留管理员专属入口（Django Admin + 用例同步）。呼应 `back-end/AGENTS.md`「保留管理员专属权限入口」。

| 角色 | 可执行操作 | 禁止操作 | 数据可见范围 |
| --- | --- | --- | --- |
| 游客（未登录） | 注册、登录 | 访问任何需鉴权接口、用例清单 | 无 |
| member（已激活普通用户，`role=member`） | 登录、登出、查看本人信息、查看用例清单 | 触发用例同步、进入 Django Admin、审批用户 | 用例清单（全部启用用例）、本人信息 |
| admin（管理员，`role=admin` 且 `is_staff=True`） | member 全部 + 触发用例同步 + 进入 Django Admin 审批/停用用户 | 无（本需求范围内） | member 全部 + Django Admin 用户管理 |

权限实现要点：
- 接口鉴权用 DRF `TokenAuthentication` + `IsAuthenticated`。
- 用例同步接口额外要求 admin：自定义权限类 `IsAdminRole`（校验 `request.user.role == 'admin'` 或 `is_staff`）。
- Django Admin 入口由 `is_staff` 控制；admin 用户须 `is_staff=True`。

## §4 功能清单与验收标准 [必填 · 核心章节]

### F1 用户注册

- **能做什么 / 程度 / 要求**：游客提交用户名、邮箱、密码完成自助注册；新用户 `status=pending`、`role=member`、`is_active=False`，**不返回 token、不能登录**，等待管理员审批。
- **关联数据表**：`accounts_user`（§6）。
- **验收标准（Given-When-Then）**：
  - `AC1.1` — Given 用户名/邮箱均未被占用 When 提交合法注册（含 `password_confirm` 一致） Then 返回 201，创建 `status=pending` 用户，响应含提示"注册成功，待管理员审批"，**不含 token**。
  - `AC1.2` — Given 用户名已存在 When 注册 Then 返回 400，提示用户名已被占用。
  - `AC1.3` — Given 邮箱已存在 When 注册 Then 返回 400，提示邮箱已被占用。
  - `AC1.4` — Given 两次密码不一致 When 注册 Then 返回 400，提示两次密码不一致。
- **异常场景**：
  - 密码不满足 Django 密码校验（过短/纯数字/常见弱密码） → 400，返回具体校验信息。
  - 邮箱格式非法 / 必填字段缺失 → 400，字段级错误。
- **边界值**：
  - 用户名长度边界（最小 3、最大 150）→ 超界 400。
  - 密码最小长度（≥8）→ 低于 400。

### F2 用户登录

- **能做什么 / 程度 / 要求**：已激活用户（`status=active`）用用户名 + 密码登录，成功返回 DRF Token 与用户信息；未激活或停用用户被明确拒绝。
- **关联数据表**：`accounts_user`、`authtoken_token`（§6）。
- **验收标准**：
  - `AC2.1` — Given `status=active` 用户提交正确密码 When 登录 Then 返回 200，含 `token` 与 `user`（id/username/email/role/status）。
  - `AC2.2` — Given 用户名不存在或密码错误 When 登录 Then 返回 400，统一提示"用户名或密码错误"（不区分两者，防枚举）。
  - `AC2.3` — Given `status=pending` 用户提交正确密码 When 登录 Then 返回 403，提示"账号待管理员审批"。
  - `AC2.4` — Given `status=disabled` 用户提交正确密码 When 登录 Then 返回 403，提示"账号已停用，请联系管理员"。
- **异常场景**：缺少字段 → 400。
- **幂等**：同一用户多次登录复用同一 Token（DRF `Token.objects.get_or_create`）。

### F3 用户登出

- **能做什么**：已登录用户登出，使当前 Token 失效。
- **关联数据表**：`authtoken_token`。
- **验收标准**：
  - `AC3.1` — Given 已登录用户 When 调用登出 Then 返回 200，删除其 Token；随后用旧 Token 访问任意鉴权接口返回 401。

### F4 当前用户信息

- **能做什么**：已登录用户获取自身资料，供前端顶部展示与权限感知导航。
- **关联数据表**：`accounts_user`。
- **验收标准**：
  - `AC4.1` — Given 有效 Token When GET `/api/auth/me/` Then 返回 200，含 id/username/email/role/status/date_joined/last_login。
  - `AC4.2` — Given 缺失或失效 Token When GET `/api/auth/me/` Then 返回 401。

### F5 管理员审批（Django Admin）

- **能做什么 / 程度**：admin 在 Django Admin 管理用户 `status`，实现审批激活与停用；`is_active` 随 `status` 自动同步（见 §5 副作用）。第一版**不做前端审批页**。
- **关联数据表**：`accounts_user`。
- **验收标准**：
  - `AC5.1` — Given `status=pending` 用户 When admin 在 Django Admin 将其 `status` 改为 `active` Then `is_active` 同步为 True，该用户随后可成功登录。
  - `AC5.2` — Given `status=active` 用户 When admin 改为 `disabled` Then `is_active` 同步为 False，该用户登录被拒（AC2.4）。
  - `AC5.3` — Given `is_staff=False` 的 member When 访问 `/admin/` Then 无法进入后台。
- **实现要点**：在 `User.save()` 或 Django Admin 动作中维护 `status → is_active` 同步规则（`active → True`，其余 `→ False`），避免两字段不一致。

### F6 用例同步（api-test 静态解析入库）

- **能做什么 / 程度**：admin 触发同步，后端用 Python `ast` 静态解析 `API_TEST_ROOT` 指向的 `test_case/` 目录，提取模块、测试函数/类、node id、docstring 标题，**upsert** 入 `testcasesnapshot`；本次未扫描到的旧用例置 `is_active=False`（软删，不物理删除）。**全程不执行 pytest**。
- **关联数据表**：`testcases_testcasesnapshot`。
- **验收标准**：
  - `AC6.1` — Given admin 已登录 When POST `/api/testcases/sync/` Then 返回 200，解析并写入快照表，响应含 `scanned/created/updated/deactivated/synced_at`。
  - `AC6.2` — Given 某用例在 api-test 中已删除 When 再次同步 Then 该用例 `is_active=False`，不物理删除；重新出现时恢复为 `is_active=True`。
  - `AC6.3` — Given member（非 admin） When POST `/api/testcases/sync/` Then 返回 403。
  - `AC6.4` — Given `API_TEST_ROOT` 指向的目录不存在 When 同步 Then 返回明确错误（4xx，业务码标识路径异常），服务不崩溃。
  - `AC6.5` — Given 用例带 `@allure.story`/`@allure.severity` When 同步 Then 对应 `story`/`severity` 字段被正确解析入库。
  - `AC6.6` — Given 用例 docstring 首行为空行、无 `@allure.title` 但有 `@allure.story` When 同步 Then `case_title` 取 `@allure.story` 值（验证标题优先级与"首个非空行"规则）。
- **解析规则**（方案 C：纯 `ast` 解析，不读源码注释）：
  - 模块 `module_key` = `test_case/` 下一级子目录名；`module_name` 第一版同 `module_key`（注释中的 `# Module Name` 不读，留后续模块配置表覆盖）。
  - 仅解析 `test_*.py` 文件中以 `test_` 开头的函数与 `Test*` 类下的 `test_` 方法。
  - `node_id` 形如 `test_case/test_xxx_case/test_xxx.py::TestClass::test_func` 或 `...::test_func`（无类时）。
  - `case_title` 来源优先级：`@allure.title(...)` > `@allure.story(...)` > **docstring 首个非空行**（注意 docstring 首行通常为空行，须跳过空白取首个非空行）；都没有则留空。
  - `story` 取 `@allure.story(...)` 值；`severity` 取 `@allure.severity(allure.severity_level.XXX)` 的 `XXX`（小写，如 critical/normal/blocker）。装饰器统一通过 `ast.FunctionDef.decorator_list` 解析。
  - **注释不解析**：`# Module Name`/`# Auth Name` 是源码注释，`ast` 不保留，本方案不读取；模块展示名/负责人留后续需求。
  - **参数化用例边界**：`ast` 静态解析无法展开 `@pytest.mark.parametrize` 的具体参数组合，第一版按"函数级"记录一条（node id 不含 `[param]` 后缀）。后续如需参数级精度，再回 api-test 阶段评估。

### F7 用例列表展示

- **能做什么 / 程度**：登录用户分页浏览启用中的用例，支持按模块与关键词筛选。
- **关联数据表**：`testcases_testcasesnapshot`。
- **验收标准**：
  - `AC7.1` — Given 已登录且快照表有数据 When GET `/api/testcases/` Then 返回 200，分页返回 `is_active=True` 用例，默认按 `module_key, node_id` 排序。
  - `AC7.2` — Given 传 `module_key` When GET Then 仅返回该模块用例。
  - `AC7.3` — Given 传 `keyword` When GET Then 对 `node_id` / `case_title` / `function_name` 做包含匹配。
  - `AC7.4` — Given 未登录 When GET `/api/testcases/` Then 返回 401。
- **边界值**：`page_size` 默认 20，最大 100；超出按 100 截断。空结果返回标准分页空集（count=0）。

### F8 Jenkins 三个独立 pipeline 重新设计（基础设施）

- **能做什么 / 程度**：将现有单一 `api-test-pipeline.groovy` 重新设计为三个相互独立的 pipeline，各自定义参数与 stage、通过环境变量调用 `ci_runner.py` 对应模式。Groovy **不复制** pytest / 重试 / node id / Allure 解析逻辑。边界 A：**仅脚本就位且可静态测试**，不联动前端触发与后端 Jenkins API。
- **关联数据表**：不涉及持久化。
- **建议产物命名**（供下游统一）：
  - `jenkins/scripts/daily-full-pipeline.groovy`（job：`api-test-daily-full`）
  - `jenkins/scripts/module-rerun-pipeline.groovy`（job：`api-test-module-rerun`）
  - `jenkins/scripts/failed-rerun-pipeline.groovy`（job：`api-test-failed-rerun`）
  - 可抽公共函数（如 `runCommand`、装环境/归档/发布 stage 模板）到 `jenkins/scripts/lib/pipeline-common.groovy`，三个 pipeline 复用，避免重复。
- **各 pipeline → ci_runner 参数映射**（以现有 `ci_runner.py` 契约为准，本需求不改 ci_runner）：
  - daily-full：跑全量。`--case-path test_case --retry-mode none`；含定时触发器（`triggers { cron(...) }`，每日凌晨，cron 值可参数化/占位）。
  - module-rerun：按模块重跑。参数 `CASE_PATH`（模块目录）；`--case-path <CASE_PATH> --retry-mode module`。
  - failed-rerun：失败重试。参数 `RETRY_MODE`（`selected`/`all-failed`）、`PYTEST_NODE_IDS`（selected 时）、`CASE_PATH`；映射 `--retry-mode selected --node-id ...` 或 `--retry-mode all-failed`。
- **验收标准**：
  - `AC8.1` — Given 三个 pipeline 已重新设计 When 运行 `python -m pytest jenkins/tests` Then 静态测试全绿。
  - `AC8.2` — Given daily-full pipeline When 静态检查 Then 含定时触发器且以全量 + `retry-mode none` 调用 ci_runner。
  - `AC8.3` — Given module-rerun pipeline When 静态检查 Then 接收 `CASE_PATH` 且以 `retry-mode module` 调用 ci_runner。
  - `AC8.4` — Given failed-rerun pipeline When 静态检查 Then 接收 `RETRY_MODE`/`PYTEST_NODE_IDS` 且以 `selected`/`all-failed` 调用 ci_runner。
  - `AC8.5` — Given 三个 Groovy When 静态检查 Then 均通过 `isUnix()` 分支兼容 Win/Linux、使用 workspace 相对路径、不含 pytest/重试/Allure 解析逻辑、不含真实凭据。
- **并发与互斥说明**：草稿要求"模块重试 Job 与失败重试 Job 相互独立、各自最多 10 并发、同模块互斥"。本需求边界 A 下，**job 独立性**通过拆分三个 pipeline 实现；**并发上限与同模块互斥**属于平台编排/Jenkins 配置层，留后续"执行链路"需求，本需求脚本不实现互斥锁。

## §5 状态机定义 [必填]

- **实体**：用户 `accounts_user.status`
- **状态全集**：`pending`（待审批） / `active`（已激活） / `disabled`（已停用）
- **不变式**：`is_active == (status == 'active')`，由模型层强制同步。
- **迁移表**：

| 源状态 | 事件 / 操作 | 目标状态 | 守卫条件 | 副作用 |
| --- | --- | --- | --- | --- |
| （无） | 用户注册 | `pending` | 用户名/邮箱唯一 | 创建用户，`is_active=False`，不发 token |
| `pending` | admin 审批通过（Django Admin 改 status） | `active` | 仅 admin | `is_active=True`，用户可登录 |
| `pending` | admin 拒绝 / 停用 | `disabled` | 仅 admin | `is_active=False` |
| `active` | admin 停用 | `disabled` | 仅 admin | `is_active=False`；建议同时删除该用户 Token（强制下线） |
| `disabled` | admin 恢复 | `active` | 仅 admin | `is_active=True` |

- 登录守卫：仅 `status=active` 可登录；`pending`/`disabled` 分别返回 AC2.3 / AC2.4 的 403。

## §7 API 契约 [必填 · 冻结后前后端共同依据]

> 统一响应包裹：成功 `{ "code": 0, "message": "...", "data": {...} }`；错误 `{ "code": <非0业务码>, "message": "...", "errors": {<字段级错误>} }`。HTTP 状态码与业务码并用。鉴权头：`Authorization: Token <token>`。

### POST `/api/auth/register/`

- **用途 / 权限**：自助注册；公开（AllowAny）。
- **请求参数**（body）：

| 参数 | 位置 | 类型 | 必填 | 校验 | 说明 |
| --- | --- | --- | --- | --- | --- |
| username | body | string | 是 | 3–150，唯一 | 登录名 |
| email | body | string | 是 | 合法邮箱，唯一 | 邮箱 |
| password | body | string | 是 | Django 密码校验，≥8 | 密码 |
| password_confirm | body | string | 是 | == password | 确认密码 |

- **成功响应**：201，`data = { id, username, email, status:"pending", role:"member" }`，message="注册成功，待管理员审批"。**不含 token。**
- **错误码**：

| HTTP | 业务码 | 含义 | 触发条件 |
| --- | --- | --- | --- |
| 400 | 1001 | 用户名已被占用 | username 重复 |
| 400 | 1002 | 邮箱已被占用 | email 重复 |
| 400 | 1003 | 两次密码不一致 | password != password_confirm |
| 400 | 1004 | 字段校验失败 | 必填缺失/邮箱非法/弱密码 |

### POST `/api/auth/login/`

- **用途 / 权限**：登录获取 Token；公开。
- **请求参数**：`username`(必填)、`password`(必填)。
- **成功响应**：200，`data = { token, user:{ id, username, email, role, status } }`。
- **错误码**：

| HTTP | 业务码 | 含义 | 触发条件 |
| --- | --- | --- | --- |
| 400 | 1101 | 用户名或密码错误 | 用户不存在或密码错误（统一提示） |
| 403 | 1102 | 账号待管理员审批 | status=pending |
| 403 | 1103 | 账号已停用 | status=disabled |

### POST `/api/auth/logout/`

- **用途 / 权限**：登出失效 Token；IsAuthenticated。
- **请求参数**：无 body（凭 Token 头识别）。
- **成功响应**：200，message="已登出"。
- **错误码**：401（未携带有效 Token）。

### GET `/api/auth/me/`

- **用途 / 权限**：当前用户信息；IsAuthenticated。
- **成功响应**：200，`data = { id, username, email, role, status, date_joined, last_login }`。
- **错误码**：401（未登录 / Token 失效）。

### POST `/api/testcases/sync/`

- **用途 / 权限**：触发 ast 静态解析同步入库；IsAuthenticated + IsAdminRole。
- **请求参数**：无 body（路径由后端 `API_TEST_ROOT` 决定）。
- **成功响应**：200，`data = { scanned, created, updated, deactivated, synced_at }`。
- **错误码**：

| HTTP | 业务码 | 含义 | 触发条件 |
| --- | --- | --- | --- |
| 403 | 1201 | 无权限 | 非 admin |
| 400 | 1202 | 用例根目录不存在/不可读 | API_TEST_ROOT 路径异常 |

### GET `/api/testcases/`

- **用途 / 权限**：用例清单分页查询；IsAuthenticated。
- **请求参数**：

| 参数 | 位置 | 类型 | 必填 | 校验 | 说明 |
| --- | --- | --- | --- | --- | --- |
| page | query | int | 否 | ≥1 | 页码，默认 1 |
| page_size | query | int | 否 | 1–100 | 每页条数，默认 20，超 100 截断 |
| module_key | query | string | 否 | | 按模块筛选 |
| keyword | query | string | 否 | | 匹配 node_id/case_title/function_name |

- **成功响应**：200，`data = { count, next, previous, results:[ { id, module_key, module_name, case_path, node_id, function_name, class_name, case_title, story, severity, is_active, synced_at } ] }`，仅返回 `is_active=True`。
- **错误码**：401（未登录）。

## §11 非功能要求 [必填精简]

- **安全**：密码 Django PBKDF2 哈希；登录失败统一提示防枚举；Token 不写入日志；停用用户即删 Token 强制下线。
- **性能**：用例列表分页（默认 20，上限 100）；同步接口对快照表按 `node_id` upsert，单次全量解析 api-test 现有规模可同步完成。
- **兼容性**：Jenkins 脚本 Win/Linux 双分支；后端时区 `Asia/Shanghai`。
- **可观测性**：同步接口返回 scanned/created/updated/deactivated 计数，便于核对。
- **已知限制**：用例同步为函数级静态解析，不展开 `parametrize` 参数组合（见 F6）；如需参数级精度，后续回 api-test 阶段评估。用例文件注释中的 `# Module Name`/`# Auth Name`（模块展示名/负责人）因 `ast` 不保留注释，本需求不读取，留后续模块配置表。
- **认证权衡**：按主人裁决（Q4）采用 DRF Token，默认无过期、无刷新机制；第一版以「登出删除 Token、停用用户强制删除 Token」控制有效期，后续如需更强安全（过期、刷新、多端会话管理）再评估升级为 JWT。
