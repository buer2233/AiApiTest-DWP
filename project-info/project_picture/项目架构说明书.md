# AiApiTest-DWP 项目架构说明书

## 1. 项目定位

`AiApiTest-DWP` 是一个面向 AI 协作和持续集成的自动化测试平台。项目正在从通用接口自动化框架升级为一套完整的 CICD 自动化测试平台，用于支持接口用例执行、失败用例重试、模块重试、Jenkins 执行查询、Allure 报告展示和 Web 页面操作。

本项目保持通用测试平台定位，不绑定任何具体业务系统。项目中的接口测试、平台接口、Jenkins 配置、Docker 配置和前端页面都应保持可迁移、可复用，不允许提交真实账号、密码、token、cookie、Jenkins API Token、生产地址或不可迁移的业务常量。

## 2. 总体架构

项目按照职责拆分为五个核心部分：

| 模块 | 目录 | 核心职责 |
|------|------|----------|
| 接口自动化执行核心 | `api-test/` | 维护接口方法、pytest 用例、失败 node id、统一 CI 执行器和 Allure 原始结果 |
| Jenkins 编排层 | `jenkins/` | 维护 Jenkinsfile 和 Groovy Pipeline，负责触发测试、归档产物和发布 Allure |
| 后端服务层 | `back-end/` | 使用 Django REST Framework 管理用户、角色、测试任务、失败用例、Jenkins 查询和报告入口 |
| 前端展示层 | `front-end/` | 使用 Vue 3 展示登录、模块通过率、失败用例弹窗、重试入口、Jenkins 入口和 Allure 报告入口 |
| 本地基础设施 | Docker Compose | 启动本地 MySQL 和 Jenkins，提供平台运行依赖 |

整体链路可以概括为：

```text
用户浏览器
  -> Vue 3 前端
  -> DRF 后端 API
  -> Jenkins 参数化构建
  -> api-test 统一执行器
  -> pytest 执行接口用例
  -> Allure 结果和运行产物
  -> Jenkins 归档与发布
  -> 后端记录任务与失败用例
  -> 前端展示通过率、失败详情、重试入口和报告入口
```

## 3. 技术栈总览

### 3.1 `api-test/`

`api-test/` 是测试执行核心，负责真正运行接口自动化用例。

主要技术：

- Python
- pytest
- requests
- allure-pytest
- pytest-rerunfailures
- Allure CLI

关键能力：

- 通过 `runpytest.py` 本地执行测试。
- 通过 `tools/ci_runner.py` 为 Jenkins 和后端提供统一执行入口。
- 通过 `tools/pytest_nodeids.py` 读取和维护 pytest node id。
- 支持模块执行、选择用例执行、一键失败重试和模块重试。
- 输出 `summary.json`、`failed_nodeids.json`、`console.log`、`allure-results/` 和 `allure-report/`。

关键目录：

```text
api-test/
├── runpytest.py
├── tools/
│   ├── ci_runner.py
│   └── pytest_nodeids.py
├── test_case/
│   └── test_*_case/
├── test_data/
├── utils/
├── report/
└── runtime/
    └── ci-runs/<run_id>/
```

### 3.2 `jenkins/`

`jenkins/` 是 CI 编排层，负责把平台参数转换为流水线执行过程。

主要技术：

- Jenkins
- Jenkins Pipeline
- Groovy
- sh
- bat
- Git
- Allure Jenkins Plugin

关键能力：

- 通过 `Jenkinsfile` 加载 `jenkins/scripts/api-test-pipeline.groovy`。
- 支持 Windows agent 和 Linux agent。
- 通过 `CASE_PATH`、`PYTEST_NODE_IDS`、`RETRY_MODE`、`RETRY_COUNT`、`CLEAN_ALLURE`、`OPEN_REPORT` 控制测试执行。
- 调用 `api-test/tools/ci_runner.py`，不在 Groovy 中复制 pytest 命令和重试逻辑。
- 归档 `runtime/ci-runs/<run_id>/` 下的执行产物。
- 发布或归档 Allure 报告。

Pipeline 阶段：

```text
Checkout
Prepare Python
Install API Test Requirements
Run API Tests
Generate Allure Report
Archive Runtime Artifacts
Publish Allure
```

### 3.3 `back-end/`

`back-end/` 是平台后端，负责平台数据、权限、测试任务、失败用例、Jenkins 集成和报告入口。

主要技术：

- Python
- Django
- Django REST Framework
- DRF Token
- MySQL
- requests
- pytest
- pytest-django

核心应用：

| 应用 | 职责 |
|------|------|
| `apps/accounts/` | 用户登录、登出、当前用户、`admin` 和 `member` 角色 |
| `apps/test_runs/` | 测试任务、失败用例、重试任务、Allure 结果解析和报告入口 |
| `apps/jenkins_integration/` | Jenkins job 查询、build 查询、console log 查询和参数化触发 |

主要 API：

```text
POST /api/auth/login/
POST /api/auth/logout/
GET  /api/auth/me/

GET  /api/test-runs/
POST /api/test-runs/
GET  /api/test-runs/{id}/
GET  /api/test-runs/{id}/failures/
POST /api/test-runs/{id}/retry-selected/
POST /api/test-runs/{id}/retry-all-failed/
POST /api/test-runs/{id}/retry-module/
GET  /api/test-runs/{id}/report/

GET  /api/jenkins/jobs/
GET  /api/jenkins/jobs/{job_name}/builds/
GET  /api/jenkins/jobs/{job_name}/builds/{build_number}/
GET  /api/jenkins/jobs/{job_name}/builds/{build_number}/console/
POST /api/jenkins/jobs/{job_name}/build/
```

核心数据模型：

| 模型 | 说明 |
|------|------|
| `User` | 平台用户，包含 `admin` 和 `member` 角色 |
| `TestRun` | 一次测试任务，记录执行参数、状态、报告路径和 summary |
| `FailureCase` | 失败用例，记录 node id、模块路径、错误类型、断言信息和重试状态 |
| `JenkinsBuildRecord` | Jenkins 触发记录，记录 job、build、参数、状态和日志入口 |

### 3.4 `front-end/`

`front-end/` 是平台操作界面，负责让用户在浏览器中完成测试任务查看和重试操作。

主要技术：

- Vue 3
- Vite
- TypeScript
- Vue Router
- Pinia
- Axios
- Element Plus
- Vitest
- Vue Test Utils

关键页面和组件：

| 页面或组件 | 说明 |
|------------|------|
| 登录页 | 用户登录，保存 DRF Token 和用户信息 |
| 平台布局 | 顶部导航、左侧菜单和主内容区域 |
| 模块通过率页 | 展示日期、用例包名、模块名、负责人、自动化负责人、通过率和运行时间 |
| 失败用例弹窗 | 展示失败用例列表、筛选条件、错误类型、执行状态和错误信息 |
| 重试操作 | 支持选择失败重试、一键失败重试和模块重试 |
| Jenkins 入口 | 打开或查看 Jenkins 任务信息 |
| Allure 报告入口 | 打开静态 Allure HTML 报告 |

前端通过 Axios 调用 DRF API，并在请求中携带 DRF Token。Pinia 负责保存登录态，Vue Router 负责路由守卫和页面跳转。

### 3.5 Docker 基础设施

Docker Compose 用于快速启动本地依赖服务，默认不把 `api-test/`、`back-end/` 和 `front-end/` 应用本身容器化。

默认服务：

| 服务 | 镜像 | 容器名 | 端口 | 数据卷 |
|------|------|--------|------|--------|
| MySQL | `mysql:8.4` | `aiapitest-mysql` | `127.0.0.1:3307 -> 3306` | `aiapitest-mysql-data` |
| Jenkins | `jenkins/jenkins:lts-jdk17` | `aiapitest-jenkins` | `8080 -> 8080`，`50001 -> 50000` | `aiapitest-jenkins-home` |

Docker 在项目中的作用：

- 为 DRF 后端提供本地 MySQL 数据库。
- 为 Jenkins Pipeline 提供本地 Jenkins 服务。
- 通过持久化数据卷保存 MySQL 数据和 Jenkins 配置。
- 通过 `docker-compose.jenkins-tools.yml` 提供可选 Jenkins 工具链镜像，该镜像可安装 Python、Git 和 Allure CLI。
- 为新环境部署提供稳定的基础设施启动方式。

## 4. 测试执行链路

### 4.1 用户从前端触发执行

1. 用户登录 Vue 3 前端。
2. 前端保存 DRF Token，并进入平台页面。
3. 用户在模块通过率页选择模块执行或重试操作。
4. 前端通过 Axios 调用 DRF 后端。
5. 后端创建或更新 `TestRun` 记录。
6. 后端根据操作类型调用 Jenkins trigger API 或登记执行结果。
7. Jenkins 收到参数化构建请求。
8. Jenkins Pipeline 调用 `api-test/tools/ci_runner.py`。
9. `ci_runner.py` 执行 pytest。
10. pytest 生成 Allure 原始结果。
11. `ci_runner.py` 输出 summary、失败 node id、控制台日志和 Allure HTML 报告。
12. Jenkins 归档产物并发布 Allure 报告。
13. 后端记录执行状态、失败用例和报告入口。
14. 前端刷新展示模块通过率、失败详情和报告入口。

### 4.2 Jenkins 直接触发执行

Jenkins 也可以由人工直接触发参数化构建。此时 Jenkins 仍然调用同一个 `ci_runner.py`，并输出同样结构的运行产物。后续平台可以通过 Jenkins 查询 API 获取 job、build、状态和日志。

### 4.3 本地命令执行

开发人员可以在 `api-test/` 下直接运行：

```powershell
python runpytest.py --case-path test_case/test_gbif_case --clean
```

也可以直接调用统一 CI 执行器：

```powershell
python -m tools.ci_runner --case-path test_case/test_gbif_case --retry-mode module --run-id local-demo --clean
```

这条链路用于本地调试和验证，不依赖前端或 Jenkins。

## 5. 失败重试设计

失败重试以 pytest node id 为核心数据结构。pytest node id 是 pytest 对某个测试用例的唯一定位字符串，例如：

```text
test_case/test_gbif_case/test_gbif_api.py::TestGbifAPI::test_species_search_by_keyword
```

平台支持三类重试：

| 重试类型 | `RETRY_MODE` | 说明 |
|----------|--------------|------|
| 选择失败用例重试 | `selected` | 用户在失败用例弹窗中选择一个或多个失败 node id 后重跑 |
| 一键失败重试 | `all-failed` | 从 pytest lastfailed cache 或失败用例记录中读取全部失败 node id 后重跑 |
| 模块重试 | `module` | 根据 `case_path` 重跑整个测试模块 |

设计原则：

- pytest 命令拼接和失败重试只在 `api-test/tools/ci_runner.py` 中实现。
- Jenkins Groovy 只传递参数，不复制 pytest 重试规则。
- DRF 后端只负责保存任务、失败用例和触发请求，不复制 pytest 重试规则。
- 前端只负责展示和发起操作，不直接接触服务器文件路径。

## 6. 报告与产物设计

每次 CI 执行会在 `api-test/runtime/ci-runs/<run_id>/` 下生成运行产物：

```text
runtime/ci-runs/<run_id>/
├── console.log
├── failed_nodeids.json
├── summary.json
├── allure-results/
└── allure-report/
```

产物说明：

| 产物 | 说明 |
|------|------|
| `console.log` | pytest 和执行器控制台日志 |
| `failed_nodeids.json` | 本次执行失败的 pytest node id 列表 |
| `summary.json` | 本次执行状态、退出码、失败用例、报告路径等摘要信息 |
| `allure-results/` | Allure 原始结果 |
| `allure-report/` | Allure HTML 静态报告 |

Jenkins 负责归档这些运行产物。后端负责记录报告路径并提供报告入口 API。前端点击 Allure 报告入口后打开静态 HTML 报告。

## 7. 数据流和控制流

### 7.1 控制流

控制流表示“谁触发谁”：

```text
用户
  -> 前端按钮
  -> DRF API
  -> Jenkins API
  -> Jenkins Pipeline
  -> ci_runner.py
  -> pytest
```

### 7.2 数据流

数据流表示“结果如何回流”：

```text
pytest
  -> allure-results / failed_nodeids.json / summary.json
  -> Jenkins 归档
  -> DRF 记录 TestRun 和 FailureCase
  -> Vue 3 页面展示通过率、失败用例和报告入口
```

### 7.3 存储流

存储流表示“数据保存在哪里”：

```text
用户、角色、测试任务、失败用例、Jenkins 触发记录
  -> MySQL

控制台日志、summary、Allure 原始结果、Allure HTML 报告
  -> api-test/runtime/ci-runs/<run_id>/
  -> Jenkins 构建归档
```

## 8. 模块边界

### 8.1 `api-test/` 边界

`api-test/` 只负责测试执行能力，包括接口方法、pytest 用例、失败 node id、重试执行器和 Allure 结果。不负责 Web 页面、数据库模型和 Jenkins stage 编排。

### 8.2 `jenkins/` 边界

`jenkins/` 只负责 Jenkins 参数、stage 编排、Windows/Linux 兼容、产物归档和 Allure 发布。不负责 pytest 重试算法和后端数据模型。

### 8.3 `back-end/` 边界

`back-end/` 只负责 DRF API、用户角色、测试任务、失败用例、Jenkins 查询触发和报告入口。不直接实现 pytest 命令拼接，不绕过 `ci_runner.py`。

### 8.4 `front-end/` 边界

`front-end/` 只负责 Vue 3 页面、登录态、模块通过率、失败用例操作、Jenkins 入口和 Allure 报告入口。不直接读取服务器文件路径，不写死真实业务模块。

### 8.5 Docker 边界

Docker 当前只负责 MySQL 和 Jenkins 基础服务。默认不容器化 `api-test/`、`back-end/` 和 `front-end/` 应用本身。

## 9. 安全与配置原则

项目所有模块都必须遵守以下原则：

- 不提交真实账号、密码、token、cookie、Jenkins API Token 或生产 URL。
- 示例配置使用占位符、环境变量或本地私有配置。
- `.env` 属于本地私有配置，不进入 git。
- Jenkins 凭据从环境变量、Django settings 或 Jenkins 凭据系统读取。
- 后端报告 API 不暴露服务器任意文件路径。
- 前端不绕过后端权限直接访问服务器文件。
- 测试用例和页面字段保持通用平台语义，不写入具体公司业务常量。
