# project-info/UI/AGENTS.md

本目录是固定 loop 第 3 阶段：UI 原型图设计阶段。进入本目录工作前，必须先遵守根目录 `AGENTS.md` 和 `project-info/AGENTS.md`，再遵守本文件。

## 阶段目标

- 默认依据 `project-info/test_case/` 的详细功能测试用例，输出 UI 原型图和交互说明。
- 如果 `project-info/demand/` 的需求说明书已冻结角色、页面目标、核心字段、状态流转、权限边界和验收口径，可以与功能测试用例阶段并行输出第一版 UI 原型。
- 原型要能指导 `front-end/` 编写 Playwright 自然语言 UI 自动化测试和 Vue 页面实现。
- 本目录只放 UI 原型资料、交互说明和原型提示词，不放前端源码。

## 技能要求

- 编写或修改 UI 原型资料时，优先使用 `prototype-prompt-generator`。
- 设计页面和交互时推荐使用 `ui-ux-pro-max`、`frontend-design`、`ckm:design-system`。
- 如果技能不可用，应记录原因，并继续按本文件的原型质量要求补齐。

## 文件命名

- 同一需求必须沿用需求阶段的名称，仅后缀不同。
- 推荐格式：`需求名-UI原型.md`、`需求名-页面交互说明.md`。
- 原型图文件使用清晰英文或拼音命名，例如 `module-pass-rate-page.png`、`failure-cases-dialog.png`。
- Figma 导出的原型图或 Codex `imagegen` 生成的原型图片必须与同一需求同名归档，例如 `需求名-dashboard-figma.png`、`需求名-failure-dialog-imagegen.png`。

## 强制设计风格基线

- UI 方案必须严格遵守仓库根目录 `DESIGN-claude.md` 中定义的前端设计风格和前端样式风格，不再仅以外部链接或口头描述作为依据。
- 原型说明必须逐项引用或转写 `DESIGN-claude.md` 的关键约束：色彩 token、字体层级、间距、圆角、状态色、组件模式、响应式策略和 Do/Don't 禁止项。
- 任何新增页面、弹窗、表格、筛选器、详情面板和状态反馈，都必须说明如何落地 `DESIGN-claude.md` 的 warm cream canvas、coral primary、dark product surface、编辑式排版和克制层级。
- 如果需求确实需要偏离 `DESIGN-claude.md`，必须在同名 UI 原型说明中写明偏离项、原因、影响范围和主人确认记录；未确认前不得进入前端开发。

## 内容要求

- UI 原型应覆盖核心页面、弹窗、表格、筛选、分页、批量操作、状态标签、加载态、空状态、错误态和权限态。
- 说明应包含页面目标、主要组件、字段来源、用户操作、状态反馈、异常处理和响应式要求。
- 管理后台设计应安静、清晰、可扫描，优先满足高频操作效率，不做营销式落地页；视觉风格必须以 `DESIGN-claude.md` 为强制基线。
- 表格、筛选器、弹窗和详情面板仍以企业管理后台效率为先；Claude 风格只作为视觉语言，不允许演变为营销页、夸张卡片堆叠、大面积渐变或装饰性插画。
- 原型说明中必须给出核心色彩、字号层级、间距、圆角、状态色和主要组件示例，功能状态色必须与暖陶土品牌强调色区分，避免成功、失败、警告状态语义混淆。
- 每个进入前端开发的 UI 方案必须至少包含一种高保真视觉产物：Figma 设计稿导出图片，或使用 Codex 系统技能 `imagegen` 生成的原型图片；仅有 Markdown 文字说明不满足 UI 阶段完成标准。
- 使用 Figma 时，应在原型说明中记录 Figma 文件/页面名称、导出图片路径、核心 frame 覆盖范围和版本时间；不得提交私有访问 token 或不可公开的组织链接。
- 使用 Codex `imagegen` 时，应在本目录保留用于生成原型图片的提示词、生成图片路径、适用页面/状态和人工校准结论，确保后续前端实现可追溯。
- 失败用例界面必须体现 `失败` 和 `跳过` 两种可展示状态，以及二者对通过率计算的差异。
- 重试操作必须体现确认、提交中、成功、失败和报告入口。
- 并行输出的 UI 原型必须标注输入来源为需求说明书，并在功能测试用例完成后补做覆盖校准。
- 覆盖校准必须检查正常场景、异常场景、边界值、权限态、状态流转、加载/空/错误状态和容器化部署下的报告入口访问方式。
- 涉及 API 数据的页面必须标注字段来源、筛选参数、分页参数、状态枚举和错误反馈，不直接假设后端字段。

## 与开发阶段的关系

- 前端开发前必须能依据本目录原型写出 Playwright 自然语言 UI 自动化测试。
- 如果开发中需要调整交互，应先更新 UI 原型和交互说明，再修改前端实现。
- 前端开发启动前必须确认本目录 UI 原型已完成测试用例覆盖校准，并且后端 API 契约已冻结。

## 安全要求

- 不提交真实账号、密码、token、cookie、租户密钥、生产 URL 或敏感地址。
- 页面示例数据使用占位符或脱敏数据，不绑定不可迁移的业务常量。
