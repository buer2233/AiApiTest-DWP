# 《需求名》可追溯矩阵（RTM）

> **作用**：把需求的每个 `AC 编号`，横向追溯到测试用例、UI、API、实现和验收状态，让机器顺着编号**自动查漏**，替代人工肉眼比对。这是 L2「跨产物一致性自动校验」的载体。
>
> **命名**：复制为 `需求名-可追溯矩阵.md`，放在 `project-info/test_case/`。
>
> **维护时机**
> - 功能测试用例阶段：建立矩阵，填「AC ↔ 测试用例」。
> - 后端/前端开发阶段：补全「实现位置」。
> - 验收阶段：核对「验收状态」，跑漂移检查清单。

## 漂移检查清单（一致性自动门禁）

> 任一项不通过即为「产物漂移」，必须回写需求 `§4`/`§12` 或补齐对应产物后才能继续。

- [ ] **无遗漏需求**：每个需求 AC 都至少有一条测试用例覆盖
- [ ] **无凭空用例**：每条测试用例都能追溯到某个 AC
- [ ] **无遗漏界面**：涉及页面的 AC 都有对应 UI 元素
- [ ] **无契约漂移**：涉及接口的 AC 都有 API 契约，且前后端实现与契约一致
- [ ] **无未实现需求**：每个 AC 都有明确实现位置
- [ ] **无孤儿代码**：实现中没有对应不上任何 AC 的功能（若有，回写需求 `§4` 或确认为越界，删除/上报）
- [ ] **全部达成**：所有 AC 验收状态为「通过」（否则不可进入验收交付）
