## 3. 功能测试用例

### 3.1 用户注册

#### TC-F-001：有效邀请码注册成功

- **关联需求**：AC1.1、AC1.5
- **优先级**：P0
- **前置条件**：系统存在状态为 `active`、未过期、未用尽的邀请码。
- **测试数据**：`username=member001`、`email=member001@example.com`、有效邀请码。
- **操作步骤**：
  1. 打开注册页。
  2. 输入邀请码、用户名、邮箱、密码和确认密码。
  3. 提交注册。
  4. 使用注册账号登录。
- **预期结果**：
  - 注册接口返回 `201 Created`。
  - 用户状态为 `active`，角色为 `member`。
  - 邀请码 `used_count` 增加 1。
  - 登录接口返回 Token 和用户信息。

#### TC-ERR-001：重复用户名或邮箱注册失败

- **关联需求**：AC1.2
- **优先级**：P0
- **前置条件**：系统已存在用户 `member001` 或 `member001@example.com`。
- **测试数据**：重复用户名或邮箱、有效邀请码。
- **操作步骤**：
  1. 使用重复用户名提交注册。
  2. 使用重复邮箱提交注册。
- **预期结果**：
  - 系统返回 `409 duplicate_user` 或字段级唯一性错误。
  - 不创建新用户。
  - 邀请码使用次数不增加。

#### TC-ERR-002：密码不一致或长度不足注册失败

- **关联需求**：AC1.3
- **优先级**：P0
- **前置条件**：系统存在有效邀请码。
- **操作步骤**：
  1. 输入两次不一致的密码提交注册。
  2. 输入长度不足 8 位的密码提交注册。
- **预期结果**：
  - 系统返回 `400 validation_error`。
  - 前端展示字段级错误。
  - 不创建用户。

#### TC-ERR-003：无效邀请码注册失败

- **关联需求**：AC1.4
- **优先级**：P0
- **前置条件**：不存在测试输入的邀请码。
- **操作步骤**：
  1. 打开注册页。
  2. 输入无效邀请码和合法用户信息。
  3. 提交注册。
- **预期结果**：
  - 系统返回 `400 invalid_invite_code`。
  - 前端在邀请码字段下展示错误。
  - 不创建用户。

#### TC-ST-001：邀请码用尽后不可继续注册

- **关联需求**：AC3.5
- **优先级**：P0
- **前置条件**：邀请码 `max_uses=1`，已被成功使用 1 次。
- **操作步骤**：
  1. 再次使用该邀请码提交注册。
- **预期结果**：
  - 系统返回 `400 invite_code_unavailable`。
  - 不创建用户。
  - 邀请码状态为 `exhausted` 或不可用。

### 3.2 登录、退出和当前用户

#### TC-F-002：正确账号密码登录成功

- **关联需求**：AC2.1
- **优先级**：P0
- **前置条件**：系统存在 `active` 用户。
- **操作步骤**：
  1. 打开登录页。
  2. 输入用户名或邮箱和正确密码。
  3. 提交登录。
- **预期结果**：
  - 登录接口返回 `200 OK`。
  - 响应包含 Token 和用户 id、用户名、邮箱、角色、状态。
  - 前端保存 Token 并跳转测试用例页面。

#### TC-ERR-004：错误密码登录失败

- **关联需求**：AC2.2
- **优先级**：P0
- **前置条件**：系统存在用户。
- **操作步骤**：
  1. 输入正确账号和错误密码。
  2. 提交登录。
- **预期结果**：
  - 系统返回 `401 invalid_credentials`。
  - 错误提示不泄露账号是否存在。
  - 不写入有效 Token。

#### TC-ERR-005：未登录访问受保护接口被拒绝

- **关联需求**：AC2.3
- **优先级**：P0
- **前置条件**：无认证 Token。
- **操作步骤**：
  1. 请求 `GET /api/v1/auth/me`。
  2. 请求 `GET /api/v1/test-cases`。
- **预期结果**：
  - 两个接口均返回 `401 unauthorized`。

#### TC-F-003：已登录获取当前用户信息

- **关联需求**：AC2.4
- **优先级**：P0
- **前置条件**：用户已登录并持有有效 Token。
- **操作步骤**：
  1. 携带 Token 请求 `GET /api/v1/auth/me`。
- **预期结果**：
  - 系统返回用户 id、用户名、邮箱、角色和状态。

#### TC-ST-002：退出登录后 Token 失效

- **关联需求**：AC2.5
- **优先级**：P0
- **前置条件**：用户已登录并持有有效 Token。
- **操作步骤**：
  1. 调用 `POST /api/v1/auth/logout`。
  2. 再次使用原 Token 请求 `GET /api/v1/auth/me`。
- **预期结果**：
  - 退出接口返回 `204 No Content`。
  - 原 Token 后续访问返回 `401 unauthorized`。

### 3.3 邀请码管理

#### TC-F-004：admin 创建邀请码成功

- **关联需求**：AC3.1
- **优先级**：P0
- **前置条件**：admin 已登录。
- **测试数据**：`code=INVITE-001`、`max_uses=5`、`expires_at=null`。
- **操作步骤**：
  1. 进入邀请码管理页。
  2. 填写邀请码、最大使用次数和过期时间。
  3. 提交创建。
- **预期结果**：
  - 创建接口返回 `201 Created`。
  - 邀请码状态为 `active`，`used_count=0`。
  - 列表刷新后展示新邀请码。

#### TC-ERR-006：member 创建邀请码被拒绝

- **关联需求**：AC3.2
- **优先级**：P0
- **前置条件**：member 已登录。
- **操作步骤**：
  1. 调用 `POST /api/v1/invite-codes`。
- **预期结果**：
  - 系统返回 `403 forbidden`。
  - 前端 member 菜单不展示邀请码管理入口。

#### TC-F-005：admin 查看邀请码列表

- **关联需求**：AC3.3
- **优先级**：P1
- **前置条件**：admin 已登录，系统存在多条邀请码。
- **操作步骤**：
  1. 进入邀请码管理页。
  2. 按状态筛选。
  3. 按关键字搜索。
  4. 切换分页。
- **预期结果**：
  - 页面展示 code、状态、最大次数、已用次数、过期时间、创建人。
  - 筛选和分页参数与 API 结果一致。

#### TC-ST-003：admin 禁用邀请码

- **关联需求**：AC3.4
- **优先级**：P0
- **前置条件**：admin 已登录，存在 `active` 邀请码。
- **操作步骤**：
  1. 在列表点击禁用。
  2. 确认操作。
  3. 使用该邀请码注册。
- **预期结果**：
  - 禁用接口返回 `200 OK`。
  - 邀请码状态变为 `disabled`。
  - 后续注册返回 `400 invite_code_unavailable`。

#### TC-E-001：邀请码边界校验

- **关联需求**：AC3.1、AC3.5
- **优先级**：P1
- **操作步骤**：
  1. 创建长度小于 6 的邀请码。
  2. 创建 `max_uses=0` 的邀请码。
  3. 创建过期时间早于当前时间的邀请码。
- **预期结果**：
  - 系统均返回 `400 validation_error`。
  - 不创建邀请码。

### 3.4 测试用例元数据同步

#### TC-F-006：admin 同步测试用例成功

- **关联需求**：AC4.1
- **优先级**：P0
- **前置条件**：admin 已登录；仓库存在 `api-test/test_case/`。
- **操作步骤**：
  1. 在测试用例页面点击同步。
  2. 等待同步完成。
- **预期结果**：
  - 同步接口返回新增、更新、缺失和总数。
  - 数据库写入可识别 pytest 用例。
  - 页面刷新后展示同步结果。

#### TC-F-007：重复同步按 node id 更新

- **关联需求**：AC4.2
- **优先级**：P0
- **前置条件**：已完成一次同步。
- **操作步骤**：
  1. 再次执行同步。
- **预期结果**：
  - 同一 node id 不重复创建。
  - 返回 `updated` 计数。
  - 总记录数稳定。

#### TC-F-008：同步提取标题和描述

- **关联需求**：AC4.3
- **优先级**：P1
- **前置条件**：测试文件存在函数 docstring 或 allure title。
- **操作步骤**：
  1. 执行同步。
  2. 查看测试用例列表。
- **预期结果**：
  - `title` 或 `description` 展示可读文本。

#### TC-ERR-007：同步源异常处理

- **关联需求**：AC4.4
- **优先级**：P1
- **前置条件**：后端配置的扫描路径不存在或为空。
- **操作步骤**：
  1. admin 调用同步接口。
- **预期结果**：
  - 源路径不存在返回明确错误。
  - 空目录返回总数 0。
  - 系统不产生未处理异常。

#### TC-ERR-008：member 同步测试用例被拒绝

- **关联需求**：AC4.1、AC3.2
- **优先级**：P0
- **前置条件**：member 已登录。
- **操作步骤**：
  1. 调用 `POST /api/v1/test-cases/sync`。
- **预期结果**：
  - 系统返回 `403 forbidden`。
  - 前端不展示同步按钮。

### 3.5 测试用例展示

#### TC-F-009：已登录用户查看测试用例列表

- **关联需求**：AC5.1
- **优先级**：P0
- **前置条件**：用户已登录，数据库存在测试用例。
- **操作步骤**：
  1. 访问 `/test-cases`。
- **预期结果**：
  - 页面展示测试用例表格和总数。
  - 表格字段包含 node id、包名、模块、路径、类名、函数名、标题、描述、同步时间。

#### TC-F-010：关键字搜索测试用例

- **关联需求**：AC5.2
- **优先级**：P1
- **前置条件**：测试用例列表存在多条数据。
- **操作步骤**：
  1. 输入函数名关键字。
  2. 输入 node id 关键字。
  3. 输入描述关键字。
- **预期结果**：
  - 页面只展示匹配数据。
  - 清空关键字后恢复列表。

#### TC-F-011：模块和包名筛选测试用例

- **关联需求**：AC5.3
- **优先级**：P1
- **操作步骤**：
  1. 选择包名筛选。
  2. 选择模块筛选。
- **预期结果**：
  - 页面只展示对应包名或模块下的测试用例。

#### TC-E-002：测试用例空状态

- **关联需求**：AC5.4
- **优先级**：P1
- **前置条件**：数据库无测试用例或筛选无结果。
- **操作步骤**：
  1. 访问测试用例列表。
  2. 输入无匹配关键字。
- **预期结果**：
  - 页面展示空状态。
  - 不显示系统错误。

#### TC-ERR-009：测试用例 API 失败时展示错误态

- **关联需求**：AC5.5
- **优先级**：P1
- **前置条件**：模拟 API 返回 500 或网络失败。
- **操作步骤**：
  1. 打开测试用例页面。
  2. 点击重试。
- **预期结果**：
  - 页面展示可恢复错误态。
  - 点击重试后重新请求数据。

#### TC-ERR-010：未登录访问测试用例页面跳转登录

- **关联需求**：AC5.6
- **优先级**：P0
- **前置条件**：浏览器无登录态。
- **操作步骤**：
  1. 直接访问 `/test-cases`。
- **预期结果**：
  - 前端跳转 `/login`。

## 5. 容器化兼容测试关注点

- 后端扫描路径必须为仓库相对路径，不使用个人本机绝对路径。
- 前端 API 地址通过 Vite 环境变量或相对路径配置，不写死宿主机端口。
- 测试数据中的账号、Token、邀请码均为占位测试值，不提交真实凭据。
