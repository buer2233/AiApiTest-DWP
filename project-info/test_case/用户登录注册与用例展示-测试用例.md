## 二、认证模块（accounts）

### F1 用户注册（AC1.1–AC1.4）

#### TC-REG-001 合法注册创建 pending 用户且不返回 token
- **关联需求**：F1 / AC1.1
- **优先级**：P0
- **测试目标**：验证合法注册创建待审批用户、响应不含 token、不可直接登录。
- **前置条件**：`<新用户名>`、`<合法邮箱>` 均未被占用。
- **测试数据**：`{ username:<新用户名>, email:<合法邮箱>, password:<强密码>, password_confirm:<强密码> }`
- **操作步骤**：
  1. POST `/api/auth/register/` 提交上述合法数据。
  2. 查询数据库 `accounts_user` 中该用户记录。
- **预期结果**：
  - HTTP 201，`code=0`，`message` 含"注册成功，待管理员审批"。
  - `data = { id, username, email, status:"pending", role:"member" }`，**响应体不含 token 字段**。
  - 数据库中该用户 `status=pending`、`role=member`、`is_active=False`。
- **备注**：与 TC-LOGIN-004 联动验证 pending 不可登录。

#### TC-REG-002 用户名已被占用
- **关联需求**：F1 / AC1.2
- **优先级**：P0
- **测试目标**：用户名重复时拒绝注册并给出明确业务码。
- **前置条件**：`<已存在用户名>` 已注册。
- **测试数据**：`{ username:<已存在用户名>, email:<另一合法邮箱>, password:<强密码>, password_confirm:<强密码> }`
- **操作步骤**：POST `/api/auth/register/`。
- **预期结果**：HTTP 400，`code=1001`，`message` 提示用户名已被占用；不创建新用户。

#### TC-REG-003 邮箱已被占用
- **关联需求**：F1 / AC1.3
- **优先级**：P0
- **测试目标**：邮箱重复时拒绝注册。
- **前置条件**：`<合法邮箱>` 已被某用户使用。
- **测试数据**：`{ username:<新用户名2>, email:<合法邮箱>, password:<强密码>, password_confirm:<强密码> }`
- **操作步骤**：POST `/api/auth/register/`。
- **预期结果**：HTTP 400，`code=1002`，提示邮箱已被占用；不创建新用户。

#### TC-REG-004 两次密码不一致
- **关联需求**：F1 / AC1.4
- **优先级**：P0
- **测试目标**：`password` 与 `password_confirm` 不一致时拒绝。
- **测试数据**：`{ username:<新用户名>, email:<合法邮箱>, password:<强密码>, password_confirm:"Mismatch#2026" }`
- **操作步骤**：POST `/api/auth/register/`。
- **预期结果**：HTTP 400，`code=1003`，提示两次密码不一致。

#### TC-REG-005 弱密码被拒（Django 密码校验）
- **关联需求**：F1 / 异常场景
- **优先级**：P1
- **测试目标**：纯数字 / 过短 / 常见弱密码触发 Django 校验失败。
- **测试数据**：分别提交 `password=<弱密码>(纯数字)`、`password="abc"(过短)`、`password="password"(常见)`，`password_confirm` 同值。
- **操作步骤**：对三组数据分别 POST `/api/auth/register/`。
- **预期结果**：均 HTTP 400，`code=1004`，`errors` 含密码字段的具体校验信息（如过短/纯数字/太常见）。

#### TC-REG-006 邮箱格式非法
- **关联需求**：F1 / 异常场景
- **优先级**：P1
- **测试目标**：非法邮箱格式被字段级校验拦截。
- **测试数据**：`{ username:<新用户名>, email:"not-an-email", password:<强密码>, password_confirm:<强密码> }`
- **操作步骤**：POST `/api/auth/register/`。
- **预期结果**：HTTP 400，`code=1004`，`errors.email` 提示邮箱格式非法。

#### TC-REG-007 必填字段缺失
- **关联需求**：F1 / 异常场景
- **优先级**：P1
- **测试目标**：缺失任一必填字段返回字段级错误。
- **测试数据**：分别缺失 `username` / `email` / `password` / `password_confirm`。
- **操作步骤**：对每种缺失分别 POST `/api/auth/register/`。
- **预期结果**：均 HTTP 400，`code=1004`，`errors` 指出对应缺失字段必填。

#### TC-REG-008 用户名长度边界
- **关联需求**：F1 / 边界值（§7 校验 3–150）
- **优先级**：P2
- **测试目标**：用户名长度下界 3、上界 150 及越界行为。
- **测试数据**：长度 2（越下界）、3（下界）、150（上界）、151（越上界）四组。
- **操作步骤**：分别 POST `/api/auth/register/`（其余字段合法、唯一）。
- **预期结果**：长度 3、150 注册成功（201）；长度 2、151 返回 400 字段校验错误（`code=1004`）。

#### TC-REG-009 密码最小长度边界
- **关联需求**：F1 / 边界值（≥8）
- **优先级**：P2
- **测试目标**：密码长度边界 8。
- **测试数据**：长度 7（如 `Ab#2026`）与长度 8（如 `Ab#20266`）两组，`password_confirm` 同值。
- **操作步骤**：分别 POST `/api/auth/register/`。
- **预期结果**：长度 7 返回 400（`code=1004`，密码过短）；长度 8 若同时满足其他复杂度则 201。

### F2 用户登录（AC2.1–AC2.4）

#### TC-LOGIN-001 active 用户正确密码登录成功
- **关联需求**：F2 / AC2.1
- **优先级**：P0
- **测试目标**：已激活用户登录返回 token 与用户信息。
- **前置条件**：存在 `status=active` 用户（用户名 `<新用户名>`、密码 `<强密码>`）。
- **测试数据**：`{ username:<新用户名>, password:<强密码> }`
- **操作步骤**：POST `/api/auth/login/`。
- **预期结果**：HTTP 200，`code=0`，`data = { token:<非空>, user:{ id, username, email, role, status:"active" } }`。

#### TC-LOGIN-002 用户名不存在（防枚举统一提示）
- **关联需求**：F2 / AC2.2
- **优先级**：P0
- **测试目标**：用户名不存在时返回统一的"用户名或密码错误"。
- **测试数据**：`{ username:"nobody_xyz", password:<强密码> }`
- **操作步骤**：POST `/api/auth/login/`。
- **预期结果**：HTTP 400，`code=1101`，`message="用户名或密码错误"`。

#### TC-LOGIN-003 密码错误（防枚举统一提示）
- **关联需求**：F2 / AC2.2
- **优先级**：P0
- **测试目标**：密码错误返回与 TC-LOGIN-002 **完全相同**的提示与业务码，确保无法据返回区分"用户不存在"与"密码错误"。
- **前置条件**：`<新用户名>` 为 active 用户。
- **测试数据**：`{ username:<新用户名>, password:"WrongPass#2026" }`
- **操作步骤**：POST `/api/auth/login/`，对比 TC-LOGIN-002 响应。
- **预期结果**：HTTP 400，`code=1101`，`message="用户名或密码错误"`，与 TC-LOGIN-002 的状态码/业务码/文案一致。

#### TC-LOGIN-004 pending 用户登录被拒
- **关联需求**：F2 / AC2.3
- **优先级**：P0
- **测试目标**：待审批用户即使密码正确也不能登录。
- **前置条件**：存在 `status=pending` 用户。
- **测试数据**：`{ username:<pending用户名>, password:<强密码> }`
- **操作步骤**：POST `/api/auth/login/`。
- **预期结果**：HTTP 403，`code=1102`，提示"账号待管理员审批"，不返回 token。

#### TC-LOGIN-005 disabled 用户登录被拒
- **关联需求**：F2 / AC2.4
- **优先级**：P0
- **测试目标**：已停用用户登录被拒。
- **前置条件**：存在 `status=disabled` 用户。
- **测试数据**：`{ username:<disabled用户名>, password:<强密码> }`
- **操作步骤**：POST `/api/auth/login/`。
- **预期结果**：HTTP 403，`code=1103`，提示"账号已停用，请联系管理员"，不返回 token。

#### TC-LOGIN-006 缺少字段
- **关联需求**：F2 / 异常场景
- **优先级**：P1
- **测试目标**：缺 username 或 password 返回 400。
- **测试数据**：分别缺 `username` / `password`。
- **操作步骤**：分别 POST `/api/auth/login/`。
- **预期结果**：HTTP 400，字段级错误提示必填。

#### TC-LOGIN-007 多次登录复用同一 Token（幂等）
- **关联需求**：F2 / 幂等（§4 F2 幂等说明，`get_or_create`）
- **优先级**：P1
- **测试目标**：同一 active 用户多次登录返回相同 token。
- **前置条件**：`<新用户名>` 为 active 用户，且未登出。
- **操作步骤**：连续两次 POST `/api/auth/login/`，记录两次返回 token。
- **预期结果**：两次返回 token 相同；`authtoken_token` 表中该用户仅一条记录。
- **备注**：若中途登出（删除 token），再次登录会生成新 token，属正常，不在本用例范围。

### F3 用户登出（AC3.1）

#### TC-LOGOUT-001 登出后旧 Token 失效
- **关联需求**：F3 / AC3.1
- **优先级**：P0
- **测试目标**：登出删除 token，旧 token 不可再用。
- **前置条件**：已登录用户持有有效 `<Token>`。
- **操作步骤**：
  1. POST `/api/auth/logout/`（头 `Authorization: Token <Token>`）。
  2. 用同一 `<Token>` GET `/api/auth/me/`。
- **预期结果**：步骤 1 返回 200，`message="已登出"`，`authtoken_token` 中该 token 被删除；步骤 2 返回 401。

#### TC-LOGOUT-002 未携带有效 Token 登出
- **关联需求**：F3 / 异常场景
- **优先级**：P1
- **测试目标**：无 token 调登出被拒。
- **操作步骤**：不带 Authorization 头 POST `/api/auth/logout/`。
- **预期结果**：HTTP 401。

### F5 管理员审批（Django Admin，AC5.1–AC5.3）

#### TC-APPROVE-001 审批激活 pending→active
- **关联需求**：F5 / AC5.1；状态机 §5
- **优先级**：P0
- **测试目标**：admin 在 Django Admin 将 pending 用户改为 active 后 is_active 同步且可登录。
- **前置条件**：存在 pending 用户；admin（is_staff=True）。
- **操作步骤**：
  1. admin 登录 Django Admin，将目标用户 `status` 改为 `active` 保存。
  2. 查询该用户 `is_active`。
  3. 该用户用正确密码登录。
- **预期结果**：`is_active=True`（与 status 同步）；登录成功返回 token（参 AC2.1）。

#### TC-APPROVE-002 停用 active→disabled
- **关联需求**：F5 / AC5.2；状态机 §5
- **优先级**：P0
- **测试目标**：admin 改 active 用户为 disabled 后 is_active 同步、登录被拒。
- **前置条件**：存在 active 用户；admin。
- **操作步骤**：
  1. admin 将该用户 `status` 改为 `disabled` 保存。
  2. 查询 `is_active`。
  3. 该用户尝试登录。
- **预期结果**：`is_active=False`；登录返回 403（`code=1103`，参 AC2.4）。

#### TC-APPROVE-003 非 staff 用户无法进入 Django Admin
- **关联需求**：F5 / AC5.3
- **优先级**：P0
- **测试目标**：`is_staff=False` 的 member 不能进入后台。
- **前置条件**：member 用户 `is_staff=False`。
- **操作步骤**：member 凭账号访问 `/admin/` 登录。
- **预期结果**：被拒绝进入后台（提示无权限/无法登录后台）。

#### TC-APPROVE-004 恢复 disabled→active
- **关联需求**：F5 / 状态机 §5 迁移表
- **优先级**：P1
- **测试目标**：disabled 用户被 admin 恢复为 active 后可重新登录。
- **前置条件**：存在 disabled 用户。
- **操作步骤**：admin 将其 `status` 改为 `active`；该用户登录。
- **预期结果**：`is_active=True`；登录成功。

#### TC-APPROVE-005 停用即删除 Token 强制下线
- **关联需求**：F5 / 状态机 §5 副作用；§11 安全
- **优先级**：P1
- **测试目标**：active 用户被停用时其现有 token 失效，强制下线。
- **前置条件**：active 用户已登录持有 `<Token>`。
- **操作步骤**：
  1. admin 将该用户改为 `disabled`。
  2. 用旧 `<Token>` GET `/api/auth/me/`。
- **预期结果**：旧 token 已被删除，步骤 2 返回 401。
- **备注**：呼应 §5"建议同时删除该用户 Token（强制下线）"；若实现以模型/Admin 动作落地，需在此验证。

### F6 用例同步（ast 静态解析，AC6.1–AC6.6）

#### TC-SYNC-001 admin 同步解析入库返回计数
- **关联需求**：F6 / AC6.1
- **优先级**：P0
- **测试目标**：admin 触发同步，正确解析并 upsert，返回统计计数。
- **前置条件**：admin 已登录；`API_TEST_ROOT` 指向含若干 `test_*.py` 的有效目录（测试用临时夹具目录）。
- **操作步骤**：POST `/api/testcases/sync/`（admin Token）。
- **预期结果**：HTTP 200，`data` 含 `scanned, created, updated, deactivated, synced_at`；`testcasesnapshot` 中按 `node_id` 写入对应记录，字段（module_key/case_path/node_id/function_name/class_name）正确。

#### TC-SYNC-002 消失用例软删与复现恢复
- **关联需求**：F6 / AC6.2；数据一致性
- **优先级**：P0
- **测试目标**：用例在源中删除后软删（is_active=False，不物理删除），重新出现时恢复。
- **前置条件**：首次同步已入库某用例 X。
- **操作步骤**：
  1. 从夹具目录移除用例 X，再次同步。
  2. 查询 X 记录。
  3. 恢复 X 文件，第三次同步。
- **预期结果**：步骤 1 后 X `is_active=False` 且记录仍存在（未物理删除），`deactivated` 计数+1；步骤 3 后 X `is_active=True` 恢复。

#### TC-SYNC-003 member 触发同步被拒
- **关联需求**：F6 / AC6.3
- **优先级**：P0
- **测试目标**：非 admin 的 member 无权同步。
- **前置条件**：member 已登录。
- **操作步骤**：POST `/api/testcases/sync/`（member Token）。
- **预期结果**：HTTP 403，`code=1201`，提示无权限。

#### TC-SYNC-004 用例根目录异常不崩溃
- **关联需求**：F6 / AC6.4
- **优先级**：P0
- **测试目标**：`API_TEST_ROOT` 指向不存在/不可读目录时返回明确业务错误且服务不崩溃。
- **前置条件**：admin 已登录；`API_TEST_ROOT` 配置为不存在路径。
- **操作步骤**：POST `/api/testcases/sync/`。
- **预期结果**：HTTP 400，`code=1202`，message 标识路径异常；进程无 500/未捕获异常。

#### TC-SYNC-005 allure story/severity 解析入库
- **关联需求**：F6 / AC6.5
- **优先级**：P1
- **测试目标**：带 `@allure.story` / `@allure.severity` 的用例正确解析。
- **前置条件**：夹具含一个用例标注 `@allure.story("登录场景")`、`@allure.severity(allure.severity_level.CRITICAL)`。
- **操作步骤**：admin POST `/api/testcases/sync/`，查询该用例记录。
- **预期结果**：`story="登录场景"`，`severity="critical"`（小写）。

#### TC-SYNC-006 标题优先级 title>story>docstring 首个非空行
- **关联需求**：F6 / AC6.6

## 七、备注与假设

1. **测试层级**：F1–F7 在后端以 pytest + pytest-django 的 API 测试落地（DRF `APIClient`）；F5 审批因走 Django Admin，可用 Admin 动作或直接模型保存 + 登录验证组合测试；F8 为 `jenkins/tests` 静态测试。
2. **夹具策略**：F6 用例同步测试使用临时夹具目录（`tmp_path` 构造若干 `test_*.py`）并指定 `API_TEST_ROOT`，不依赖 `api-test/` 真实用例数量，保证可重复与隔离。
3. **统一响应**：所有断言基于 §7 统一响应包裹（成功 `code=0`；错误非 0 业务码 + `errors`）与对应 HTTP 状态码。
4. **安全**：测试数据均为占位/示例值；token 运行时动态获取，不硬编码；不出现真实凭据与生产地址。
5. **不在本阶段验证**：并发上限、同模块互斥锁（§4 F8 说明属后续执行链路需求）；JWT/刷新/过期（§11 认证权衡，本版不做）。
