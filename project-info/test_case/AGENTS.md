# project-info/test_case/AGENTS.md

本目录是固定 loop 第 2 阶段：功能测试用例阶段。进入本目录工作前，必须先遵守根目录 `AGENTS.md` 和 `project-info/AGENTS.md`，再遵守本文件。

## 阶段目标

- 依据 `project-info/demand/` 的详细需求说明书，输出详细功能测试用例，也就是详细设计。
- 测试用例必须能反向验证需求功能、数据规则、状态流转和验收口径。
- 本目录只放测试设计资料，不放 `api-test/` 中可执行 pytest 用例代码，也不放 Playwright 代码。

## 技能要求

- 编写或修改测试用例资料时，优先使用 `test-cases`。
- 设计涉及接口契约时，可结合 `api-design` 的资源和错误模型原则。
- 如果技能不可用，应记录原因，并继续按本文件的测试用例质量要求补齐。

## 文件命名

- 同一需求必须沿用需求阶段的名称，仅后缀不同。
- 推荐格式：`需求名-测试用例.md`。
- 回归清单可使用：`需求名-回归测试清单.md`。

## 内容要求

- 测试用例按模块和优先级组织。
- 每条用例应包含用例编号、关联需求、测试目标、优先级、前置条件、测试数据、操作步骤、预期结果和备注。
- 必须覆盖正常场景、异常场景、边界值、权限边界、状态流转和数据一致性。
- 涉及失败用例状态时，必须覆盖 `失败`、`通过`、`跳过`、`不展示` 的展示、修改和统计影响。
- 涉及执行链路时，必须覆盖 Jenkins 触发、运行中、成功、失败、重试、报告归档和同步异常。

## 与开发阶段的关系

- 后端接口 pytest 测试应依据本目录用例拆分。
- 前端 Playwright 自然语言 UI 自动化测试应依据本目录用例和 `project-info/UI/` 原型拆分。
- 如果实现阶段发现测试用例缺口，应先回到本目录补充，再继续开发。

## 安全要求

- 测试数据使用占位符或脱敏数据。
- 不写入真实账号、密码、token、cookie、租户密钥、生产 URL 或敏感地址。
