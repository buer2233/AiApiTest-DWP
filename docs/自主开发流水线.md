# 自主开发流水线（Autonomous Dev Pipeline）

> 本文件是 `AiApiTest-DWP` 固定开发 loop 的**自动化执行层总纲**。它不替代各阶段 `AGENTS.md`，而是说明"需求冻结后，如何让开发流程尽量自主跑完，主人只守需求和验收两头"。
>
> 一句话目标：**主人深挖需求 → 冻结 → 中间自主流水线跑完 → 主人验收。**

---

## 1. 核心理念

- **可执行规格驱动**：下游自动化的可靠性 = 需求规格的完整度。把决策前置到需求阶段，让开发阶段无需猜。需求文档用 `demand/_需求说明书模板.md`（executable spec）。
- **两头人工、中间自动**：人的认知投入集中在"定义正确的问题"（需求冻结）和"判断结果对不对"（验收），中间执行交给 agent。
- **门禁刚性、产物弹性**：流程深度可按需求分级裁剪（产物），但质量门禁任何档位都不可裁剪。
- **禁止脑补、撞墙熔断**：无人值守 ≠ 遇到不确定就硬猜。撞到规格未覆盖的关键决策，暂停上报主人，绝不脑补。

## 2. 自动化成熟度阶梯（L0–L3）

| 级别 | 能力 | 主人体感 |
| --- | --- | --- |
| **L0** 现状起点 | 人工驱动，每阶段停下等指示 | "我得一路盯着指挥" |
| **L1** 需求可执行化 | executable spec 模板 + 交互式深挖 + 阶段自动衔接 | "需求冻结后它能自己跑很远，只在撞墙时问我" |
| **L2** 自动门禁 | 独立对抗审查 + RTM 一致性校验 + 验收包聚合 | "中间质量机器把守，验收有一站式证据" |
| **L3** 一键编排 | Workflow 脚本化整条流水线，断点续跑 | "提需求 → 喝咖啡 → 验收包" |

> 当前建设目标：**L1 + L2**。L3 待 L1/L2 跑顺后再上。

## 3. 人机介入模型（从"每步审批"到"守两头 + 熔断"）

```text
需求深挖[主人深度参与]
   └─ 我主导提问，逐项填满 executable spec，消灭脑补点
      ↓
✅ 卡点① 需求冻结[主人签字]  ← §0 待澄清闭环 + 架构/契约/容器化门禁通过
      ↓
〔自主流水线：功能用例 → UI 原型 → 后端 TDD → 前端 TDD〕
   └─ 机器把守门禁；撞到规格盲区/门禁失败 → ⚡熔断，一次性批量问主人
      ↓
✅ 卡点② 最终验收[主人终审]  ← 依据自动聚合的「验收包」
```

主人只在**两个卡点 + 偶发熔断**介入。需求挖得越透，熔断越少，理想 ε→0。

## 4. 需求冻结后的自动衔接

冻结后各阶段按依赖自动接力，无需主人逐阶段点头（依据根 `AGENTS.md` 并行规则）：

1. **功能测试用例**：依据 §4 的 AC 编号逐条转用例，建立 `需求名-可追溯矩阵.md`（RTM）。
2. **UI 原型**（涉及页面时）：依据 §8 UI 字段级规格输出，完成后与测试用例做覆盖校准。
3. **后端 TDD**：API 契约已随需求冻结 → 先写 pytest 接口测试 → 实现 → 回归；补全 RTM 实现列与证据。
4. **前端 TDD**：依据 UI 原型 + 冻结契约 → 先写 Playwright → 实现 → 回归；补全证据。
5. **聚合验收包**：回归通过后生成 `需求名-验收包.md`，交主人终审。

## 5. 三道自动质量门禁（替代原人工门禁）

| 门禁 | 机制 | 替代了 |
| --- | --- | --- |
| **功能正确性** | TDD（pytest / Playwright 退出码作客观 pass/fail）+ 证据物留存 | 人工功能确认 |
| **独立对抗审查** | 开发完成后由**独立 review subagent** 挑刺（`receiving-code-review`/`security-review`），做工的不给自己打分 | 人工代码审查 |
| **跨产物一致性** | RTM 漂移检查清单：需求↔用例↔UI↔契约↔实现↔验收 顺 AC 编号自动查漏 | 人工一致性核对 / UI 覆盖校准 |

## 6. 验收包交付

验收阶段主人只看一份 `需求名-验收包.md`，含：逐条 AC 验收结论、测试证据（pytest/覆盖率/Playwright/截图/Allure）、RTM 一致性摘要、实现摘要、**待主人确认项（熔断遗留）**。主人签字「通过」或「打回」。

## 7. 熔断上报协议

agent 在以下情况**必须暂停并一次性批量上报主人**，不得脑补：

- 需求规格（含 §0）未覆盖的关键决策点；
- 任一质量门禁失败且无法自动修复；
- 产物间出现无法自动调和的冲突（如契约与实现矛盾）；
- 触及权限、安全、数据销毁、对外发布等高风险操作。

上报的问题及主人裁决，必须回写需求文档 §0/§13 与验收包第 6 节，形成可追溯闭环。

## 8. 监督式起步策略

不一上来就全放开无人值守。**前 2–3 个需求采用"监督式自动跑"**：agent 连续推进，但每个大阶段（用例/后端/前端完成）留一个轻量 checkpoint 让主人瞄一眼。校准信任后，再逐步放开为真正无人值守。

## 10. 与固定 loop 的关系

本流水线是固定 loop 的**执行方式说明**，不改变 loop 的阶段构成与产物归属：

- 阶段、目录、命名、产物 → 仍以根 `AGENTS.md` 和各子目录 `AGENTS.md` 为准。
- 本文件只回答"这些阶段如何尽量自主衔接、质量如何机器把守、人在哪两头介入"。
- 二者冲突时，以根 `AGENTS.md` 的流程与门禁定义为准。
